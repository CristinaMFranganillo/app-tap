Contabilidad — Enfoques de implementación

Cómo estructurar el componente y los datos

A

Componente único con lógica interna

Un solo ContabilidadComponent que maneja las 3 vistas (día/mes/año) con tabs, similar a cómo funciona CajaComponent actualmente.

Pros

  • Patrón ya probado en Caja
  • Menos archivos, navegación simple
  • Estado compartido entre vistas

Cons

  • Componente grande (3 vistas + CRUD + formulario)
  • Un solo archivo .ts con mucha lógica

⭐ Recomendado — Consistente con el patrón de Caja. La lógica extra (CRUD manuales) es manejable con signals.

B

Componente padre + sub-componentes por vista

Un ContabilidadComponent shell con 3 componentes hijos: ContabilidadDiaComponent, ContabilidadMesComponent, ContabilidadAnioComponent.

Pros

  • Archivos pequeños y enfocados
  • Cada vista es independiente

Cons

  • Más archivos (4 componentes mínimo)
  • Comunicación padre-hijo para navegación drill-down
  • Rompe el patrón establecido por Caja
C

Rutas independientes por vista

Rutas separadas: /admin/contabilidad/dia, /admin/contabilidad/mes, /admin/contabilidad/anio. Cada una con su componente lazy-loaded.

Pros

  • URLs compartibles y con deep-link
  • Componentes totalmente aislados

Cons

  • Pierde estado al cambiar de vista (recarga datos)
  • Navegación drill-down más compleja con query params
  • Complejidad innecesaria para una PWA mobile-first
